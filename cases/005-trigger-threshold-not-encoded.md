# Case 005 — 코드화 트리거 임계점이 정성적으로만 존재 (자동화 지연 역설)

> 이 케이스는 **메타 케이스**다. 사건 발원지는 ai-devops-orchestrator 자신이다.
> case #003이 캡처 채널을 박제하기 위한 채널을 만들었듯, 이 케이스는 **케이스 → 코드 변환의 임계점을 박제하기 위한 절차**를 만든다.

## 1. 무슨 일이 있었나 (사실)

- 2026-05-02, ai-devops-orchestrator의 진행 상태에 대한 메타 점검(컨설팅 톤) 중, 다음 두 가지가 명확해짐:
  - case #002의 후보 B(`repository_dispatch` 자동 dispatch)와 후보 C(주기적 sweep)는 "다음 실제 트리거 발생 시"라는 정성적 표현으로 보류 중
  - case #004(가드 비대칭 패턴)도 "두 번째 인스턴스 누적까지 코드화 보류"라는 정성적 표현으로 observing 상태
- 즉 **5개 케이스 박제 / 코드화 PR 0건**이 두 메커니즘으로 누적될 수 있는 구조:
  1. 첫 번째 인스턴스가 잡혔을 때 임계점 룰이 없으니 "두 번째 기다림"이 자연스러운 결정이 됨
  2. 두 번째 인스턴스가 잡혀도 "충분히 강한가?"를 사람이 매번 다시 판단해야 함
- 사용자 제안: "케이스가 특정 개수(예: 유사 인스턴스 2회 발생)에 도달했을 때 코드화를 강제하는 하드 리미트 룰을 추가하자"
- Claude Code 변형 권고: 강제 대상을 **"코드 작성"이 아니라 "결정 절차 진입"** 으로 — 결정권은 승인자(사용자)에게 남기되, 회피 경로 차단

## 2. 어느 액터/레이어에서 발생했나

- **레이어가 아니라 절차** — 케이스 파이프라인 자체의 메타 결함
- 영향 범위: L1~L5 모두. 트리거 룰이 없으면 어느 레이어도 코드화 시점이 도래하지 않음
- 액터 경계 측면: **"코드화 결정은 사용자 영역, 결정 강제는 orchestrator 영역"** 의 분할이 필요

## 3. 오케스트레이터가 막거나 도울 수 있었는가

이건 패턴 결함이 아니라 절차 결함이므로, 막는 것이 아니라 **절차 추가**가 답:

- [x] 임계점 룰: 동일 패턴 인스턴스 **2회 누적 시 결정 케이스 발행 의무**
- [x] 결정 케이스 양식 정의 (이 파일 §6 참조)
- [x] 결정 마감일 정책 (기본 7일)
- [ ] 자동화: 결정 케이스 발행 자체를 자동 트리거(예: case 인덱스 메타데이터 기반 GitHub Action)는 후속 케이스에서 — 본 PR에서는 절차/양식만 박제

## 4. 반복 가능성

- **매우 높음** — 가드 비대칭 외에도 마이그레이션, 환경변수, 라우트 검증 등 여러 도메인에서 동일하게 "한 번 보고 두 번 기다리는" 패턴이 반복됨

## 6. 결정 케이스(Decision Case) 양식

결정 케이스는 일반 케이스의 변형이며, 다음 frontmatter 추가 필드를 가진다:

```yaml
---
id: NNN
title: <패턴 이름> — 코드화 결정 케이스
date: YYYY-MM-DD
project: ai-devops-orchestrator
actor_involved: [user, orchestrator]
state: pending-decision  # → decided-implement | decided-observe | decided-wontfix
deadline: YYYY-MM-DD     # 발행일 + 7일 (기본). 연장은 본문에 명시 + 코멘트 근거
related_cases: [NNN, NNN]  # 누적된 인스턴스 케이스 ID
---
```

본문 섹션:

1. **누적된 인스턴스** — 각 인스턴스 케이스 ID + 한 줄 요약
2. **공통 패턴** — 인스턴스들이 공유하는 도메인/레이어/룰
3. **어느 레이어에서 막을 수 있는가** — L1~L5 중 단수/복수
4. **옵션** — 다음 셋 중 하나를 선택해야 함:
   - **(a) 코드화** — 어느 레이어, 무엇을, 어떤 PR로 (구체적 범위)
   - **(b) 추가관찰** — 다음 인스턴스까지 보류 (왜, 무엇을 더 봐야 하는가)
   - **(c) wontfix** — 코드화 비용이 가치 초과 (왜)
5. **사용자 결정** — 위 셋 중 체크 + 결정자 + 결정일
6. **마감일 도래 시 자동 처리** — 결정 없이 deadline 경과 → 본 케이스를 코멘트로 회고하는 후속 케이스 자동 발행 의무 (현재 수동 — 향후 자동화 후보)

## 8. 학습 (ChromaDB 인덱싱 대상)

케이스 주도 원칙은 **자기 자신에게도 적용된다**. "임계점 룰이 부재하다"는 것 자체가 케이스가 되어야, 임계점 룰이 추가될 수 있다 — 그렇지 않으면 "케이스 주도 원칙을 자동화하기 위해 가설 기반 룰을 추가하는" 자기 모순이 발생한다. 결정 강제는 **"코드 작성 강제"가 아니라 "결정 절차 진입 강제"** 로 표현될 때만 액터 경계를 보존한다 — 코드화 결정권은 승인자(사용자)에게 남고, orchestrator는 사용자가 결정을 내리도록 강제한다. 이 분할이 자동화 지연 역설(수동 대행이 매끄러우면 코드화 압력이 사라진다)을 깨는 메커니즘이다.
