# Case 006 — 미정의 도메인 인보리언트 (Undeclared Domain Invariant, UDI)

> **가드 비대칭(case #004)과는 별개의 패턴이다.**
> 가드 비대칭은 "형제 라우트 중 한쪽에는 가드가 있는데 다른 쪽에 없음"이고,
> UDI는 "도메인 룰 자체가 코드·스키마 어디에도 선언된 적이 없음"이다.

## 1. 무슨 일이 있었나 (사실)

- 2026-05-02, case #005 절차의 **첫 자기 검증** 중 발견됨
- case #004가 "reviews 도메인의 두 번째 가드 비대칭 후보"로 인용한 gwangcheon-shop **#372**(리뷰 중복 작성 방지)를 트리거 시점에 재검증
- 검증 데이터:
  - `src/app/api/admin/reviews/route.ts` — `GET`만
  - `src/app/api/admin/reviews/[id]/route.ts` — `PATCH`, `DELETE`만
  - **admin 측에 review POST가 없음** → 관리자는 리뷰를 생성하지 않음 → "비대칭"이 성립할 비교 대상 자체가 부재
  - `prisma/schema.prisma:208` — 현재는 `@@unique([userId, productId])`가 적용됨 (#372 머지 후)
- **결과**: #372는 가드 비대칭이 아니라 본질이 다른 결함:
  - 스키마 단계 — `@@unique` 부재
  - 라우트 단계 — POST에 중복 체크 부재
  - 즉 "상품 1건당 1리뷰"라는 도메인 룰이 **schema와 route 어디에도 처음부터 선언된 적이 없었음**
- #372는 이미 **CLOSED** (해결됨). 본 케이스는 그 결함의 패턴 카테고리를 박제

## 2. 어느 액터/레이어에서 발생했나

- **Claude Code (생성자)**: 도메인 모델/라우트 작성 시 인보리언트를 코드·스키마에 박제하지 않음 — 룰이 머릿속에만 있다가 누락
- **Antigravity (검증자)**: "이 도메인의 인보리언트가 명문화됐는가?"를 점검하는 패턴 부재
- **orchestrator (지휘자)**:
  - L3 컨텍스트 패킹이 "이 모델이 다루는 도메인의 알려진 인보리언트 목록"을 검증 컨텍스트에 포함하지 못함
  - L4 검증 게이트가 새 모델/라우트 추가 시 인보리언트 명문화 여부를 점검하지 못함

## 3. 오케스트레이터가 막거나 도울 수 있었는가

- [x] L1 (컨텍스트 주입) — 부분적. 생성자 프롬프트에 "도메인 인보리언트 카탈로그 참조" 힌트
- [ ] L2 (트리아지) — 라우팅 문제 아님
- [x] L3 (컨텍스트 패킹) — **가장 강력**. 도메인 인보리언트 카탈로그(`docs/DOMAIN_INVARIANTS.md`)를 검증 컨텍스트에 포함
- [x] L4 (검증 게이트) — 새 모델/스키마 추가 PR에서 카탈로그 갱신 여부 점검
- [ ] L5 (승인·배포) — 너무 늦음
- [ ] 어디서도 못 막음 — 해당 없음

> 단독 코드화는 case #005 절차에 따라 **두 번째 UDI 인스턴스가 누적될 때까지 보류**한다.

## 4. 반복 가능성

- **매우 높음** — 도메인 모델이 추가될 때마다 인보리언트 명문화 누락 가능
- **영향**: 큼 — 데이터 무결성, 비즈니스 룰 일관성

### 다음 인스턴스 발견 방법

- 추적 단서: 새 도메인 모델/스키마 추가 PR에서 "이 모델의 인보리언트가 어디 박혀있나?"를 점검하는 자기 질문이 사실상 유일한 접근

## 5. 식별 기준 (가드 비대칭과의 MECE 구별)

세 기준 모두 yes일 때만 UDI:

| # | 기준 | UDI | 가드 비대칭 |
|---|---|---|---|
| ① | 결함의 본질이 "특정 도메인 룰이 코드·스키마 어느 곳에도 명시되지 않음"인가? | **yes** | no |
| ② | 같은 룰을 다루어야 할 형제 모듈이 부재하거나, 존재해도 함께 부재한가? | **yes** | no (한쪽엔 존재) |
| ③ | 수정이 "한쪽에 가드 추가"가 아니라 "룰을 어디에 어떻게 박을지 결정"부터 시작해야 하는가? | **yes** | no (위치는 자명) |

세 가지 중 하나라도 no면 다른 카테고리(가드 비대칭, 단순 버그, 컨텍스트 부족 등). 트리거 시점 재검증의 표준 체크리스트.

## 8. 학습 (ChromaDB 인덱싱 대상)

가드 비대칭(case #004)과 UDI(case #006)는 표면 증상이 비슷해 보이지만 **검증 컨텍스트의 종류가 다르다**. 가드 비대칭의 방어선은 "형제 라우트 번들"(같은 도메인을 다루는 라우트 집합)을 검증 컨텍스트로 묶는 것이고, UDI의 방어선은 "도메인 인보리언트 카탈로그"(룰 → 박제 위치 매핑)를 검증 컨텍스트에 포함시키는 것이다. 같은 L3 컨텍스트 패킹의 책임이지만, **두 가지 다른 입력 데이터셋**이 필요하다. 따라서 두 패턴을 한 코드화 PR로 묶지 말고 별도로 진행해야 한다 — 첫 인스턴스만 박힌 시점에서 강제로 일반화하면 두 패턴이 서로의 노이즈가 된다.

또한 본 케이스는 case #005 절차의 가치를 사후적으로 입증한다. case #004가 "두 번째 인스턴스 후보"로 #372를 가볍게 인용했을 때 정의 일치 여부를 재검증하지 않았던 것이, 트리거 시점에 비로소 드러났다. **후보 박제와 트리거 박제는 다른 단계의 활동**이며, 후자에는 식별 기준 표(§5 같은) 적용이 의무화되어야 한다.
