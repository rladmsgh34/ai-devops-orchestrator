# Case 002 — 케이스 캡처 채널 부재

## 1. 무슨 일이 있었나 (사실)

- 2026-05-01, case #001을 마무리한 직후 사용자가 "현재 프로세스에 문제점은 없을까?"를 물음
- 점검 결과 **가장 큰 위험**으로 식별된 항목: 케이스 주도 원칙은 정의됐지만, 실제 사건(빌드 실패, 검증 누락, 운영 사고, 사용자 피드백)이 어떻게 `cases/`로 흘러오는지 채널이 정의되지 않음
- 현재까지 등록된 케이스 #000(메타), #001(자기 점검), #002(이 케이스) — 모두 사용자가 즉석에서 발견하고 손으로 작성. **gwangcheon-shop 운영에서 발견된 사건이 자동/반자동으로 흘러오는 경로는 0개**

## 2. 어느 액터/레이어에서 발생했나

**Layer 2 — 트리아지 라우팅** 의 가장 앞단. ARCHITECTURE.md에 "알림(에러 로그, 알람, 사용자 피드백)이 들어오면 분류해 올바른 액터에게 보냄"이라고 적었지만, **알림이 어디서 들어오는가**가 비어있음.

## 3. 오케스트레이터가 막거나 도울 수 있었는가

해당 없음 (메타 케이스). 단, 이 케이스가 풀리지 않으면 **케이스 주도 원칙 자체가 데드락**에 빠진다:

- 사용자가 매번 손으로 발견·기록 → 시간이 지날수록 누락·왜곡
- 누락된 사건 = 누락된 컴포넌트 = 죽은 척추
- 결국 "케이스가 없어서 코드도 안 짜는" 정체

## 4. 반복 가능성

- 지속적 운영 중 매일 발생: 빌드 실패, 검증 실패, main 직접 푸시 시도, 운영 알림
- 채널이 없으면 **모두 누락**됨
- 영향: 결정적 (오케스트레이터의 존재 의의)

## 5. 결정 — 캡처 채널 후보

이 케이스는 **결정 케이스**다. 코드 변경 없이, 다음 PR(또는 후속 케이스)이 어떤 채널을 코드화할지 결정한다.

### 후보 A — GitHub 이벤트 수동 발행 (최소)
- gwangcheon-shop의 PR/Actions가 실패할 때 **사용자가** 이 저장소에 issue를 한 줄 등록
- 오케스트레이터는 issue → `cases/NNN-*.md` 변환 가이드만 제공
- 비용: 0 / 누락 위험: 큼

### 후보 B — GitHub Actions cross-repo 디스패치 (반자동)
- gwangcheon-shop의 워크플로우가 특정 실패 조건(검증 누락 PR 머지, main 직접 푸시 차단, 운영 헬스체크 실패)에서 이 저장소에 `repository_dispatch` 이벤트 발행
- 오케스트레이터에 그 이벤트를 받아 `cases/NNN-*.md` 초안을 issue로 만드는 워크플로우 추가
- 비용: 워크플로우 1개씩 양쪽에 / 누락 위험: 사전 정의한 트리거에 한정됨

### 후보 C — 주기적 sweep (백업)
- 오케스트레이터가 주 1회 gwangcheon-shop의 마지막 N개 PR/run을 점검 → 룰 위반·검증 누락이 있으면 케이스 초안 발행
- 후보 A/B를 보강하는 안전망
- 비용: 스케줄러 + GitHub API 토큰 / 누락 위험: 트리거 빈도에 따른 지연

### 권장
- **후보 A를 즉시 도입** (GitHub issue 템플릿만 추가하면 됨, 코드 0)
- **후보 B는 첫 사례가 발생한 후** 그 사례를 케이스로 박제하면서 함께 코드화
- 후보 C는 후보 B가 누락하는 패턴이 발견되면 추가

## 7. 학습 (ChromaDB 인덱싱 대상)

케이스 주도 원칙은 **케이스가 들어오는 경로**가 정의되어 있을 때만 작동한다. 새 정책을 도입할 때 "이 정책이 작동하려면 어떤 입력이 필요하고, 그 입력은 어디서 들어오는가"를 함께 정의하지 않으면 정책 자체가 빈 껍데기로 남는다. 케이스 발행은 **사용자의 호의**가 아니라 **시스템의 기능**이어야 한다.
