# Case 011 — 트리거 발화 메커니즘 부재 (결정 케이스, family 4차)

**TL;DR.** 사용자가 본 레포 자기 비판을 요청 → claude-code가 8개 실측 문제 중 "발화 메커니즘 부재(B)"를 가장 시급한 항목으로 추천 → 사용자가 "추천 진행해줘"로 코드화 진입을 신호. 이는 case #010 §8이 예언한 family 4차 형태("결정 본문이 룰 인용만으로 완전 도출되는 단계")의 첫 발생점. 곧장 코드화하면 case #007/009/010 family 4차 인스턴스를 *발생점에서 의도적으로 발생*. 결정 추천은 (b) `decided-observe` — case #010 §1.1 "발생 직전 발견 → 결정으로 회피" 표준 경로의 4차 자동 적용. 본 PR 코드 0줄.

> 본 case 본문은 case #010 §1.1·§8의 직접 인용으로 구성된다 (case #010 §8 예언의 첫 검증). 본문 길이가 case #010 대비 짧을수록 예언이 검증된다.

## 1. 입력 — 추천에 대한 추천

직전 대화 (2026-05-05):
- 사용자: "ai devops 의 문제점은 뭐가 있을까"
- claude-code: 8개 실측 문제 → 핵심 두 가지(A 외부 사건 캡처 부재, B 발화 메커니즘 부재) → "B가 A보다 먼저"
- claude-code 자기 메모(직전 답변 마지막 줄): "이것도 *지금 발생한 사건*이 발생해야 케이스화하는 게 본 레포 룰입니다"
- 사용자: "추천 진행해줘"

입력 본질: case #010 §1과 동일 — *가설 단계 추천*이 코드화 신호로 작동. 사용자가 마찰을 *경험*한 사건이 아니라, claude-code의 분석 결과 추천이 입력.

## 2. 자기 검증 — case #007/009/010 family의 4차 인스턴스인가?

case #010 §1.1이 case #007/009 두 인스턴스에 대해 한 자기 검증을 본 case가 case #007/009/010 세 인스턴스에 대해 수행한다.

| 차수 | 케이스 | 발견 시점 | 결정 비용 | 회수 비용 |
|---|---|---|---|---|
| 1차 | #007 | 박제 후 활동 정지 시점 | 룰 발행 (case #005 §6 정의) | archive 4개 repo |
| 2차 | #009 | 박제 후 외부 호출 0회 식별 | 룰 적용 | profile 격리 5개 컨테이너 |
| 3차 | #010 | 박제 행위 직전 (코드 0줄) | 룰 자동 인용 | 0 (회수할 자산 없음) |
| **4차** | **본 case** | **추천 → 코드화 신호 사이 (추천 본문만 존재)** | **룰의 자기 코드화 후보** | **0** |

4차 형태 (case #010 §8 예언):
> "다음 자기 적용(4차)은 결정 본문이 룰 인용만으로 완전 도출되는 단계가 될 후보 — 그 시점이 결정 케이스의 자동화 진입점"

본 case 본문 §3·§4·§6은 case #010 §1.1·§8과 case #005 §6 인용으로 구성됨. 박제 비용이 0에 가까움 (예언의 첫 실측 검증).

## 3. 발화 메커니즘 부재 — 실측

> 본 §3은 *증거*만 박제. 어느 한 줄도 본 PR로 코드화되지 않음.

### 3.1 정의된 룰 vs 발화 메커니즘

| 룰 (정의됨) | 발화 메커니즘 (현재) |
|---|---|
| case #005 §"마감일 도래" — deadline 경과 시 회고 케이스 발행 | **수동** (case #005 본문 명시: "현재 수동, 향후 자동화 후보") |
| case #005 §임계점 — 동일 패턴 2회 누적 시 결정 케이스 발행 의무 | **수동** — 능동 감지 부재 |
| case #004 가드 비대칭 2회차 누적 시 활성화 | 메모리 [admin/user 가드 대칭성 점검] 1건이 유일 채널 |
| case #006 UDI 2회차 누적 시 활성화 | 동일 — 메모리 1건 |
| case #010 §6 트리거 임계점 3종 | 모두 수동 |

룰 5종 × 발화 채널 1종(claude-code 메모리) = **4종 룰이 발화 보장 없이 박제됨**.

## 4. 어느 레이어/액터에서 발생했나

case #007/009/010과 동일하게 **레이어 결함이 아닌 컴포넌트 발행 절차의 결함 후보** family에 속함.

- 직접 책임: 사용자(추천 진행 신호) + claude-code(추천 생성)
- 메타 책임: orchestrator가 case #010 §8 예언을 자동 인용해 진입을 결정 케이스로 변환
- L1~L5 어느 레이어에 매핑되지 않음 — 컴포넌트 발행 절차의 메타 박제

## 5. 옵션 (3지선다)

### (a) `decided-implement` — 발화 메커니즘 즉시 코드화

후보 컴포넌트 (3종):
1. **deadline cron** — GitHub Actions cron 워크플로우. `cases/*.md` frontmatter `deadline` 필드 검사, 도래 시 회고 issue 자동 발행
2. **dispatch (case #002 후보 B)** — 광천샵 PR 머지 시 특정 라벨/도메인이면 본 레포에 case 후보 issue dispatch
3. **패턴 정적 감지** — admin/user 가드 누락·UDI 식별을 광천샵 PR diff에서 정적 감지 (case #002 후보 C 변형)

비용:
- family 4차 인스턴스를 *의도적으로 발생*. case #010 §1.1 "회수 비용을 0에 가깝게 만든다(회수할 자산이 아직 없으므로)"의 직접적 위반 — 회수할 자산 3+개 발생.
- case #009·#010이 박제한 "케이스 없이 컴포넌트를 박제하지 않는다" 룰의 첫 자기 위반.

가능성:
- 사용자가 case #010 §6 트리거 임계점이 *과도한 보수*임을 결정한 경우. 현 시점 그 신호 없음.

### (b) `decided-observe` — 박제만, 첫 발화 누락 사건까지 보류 ⭐ (claude-code 추천)

- §3 실측 + §7 트리거 임계점만 본 case에 박제. 본 PR 코드 0줄.
- case #010 §1.1 "발생 직전 발견 → 결정으로 회피" 표준 경로의 4차 자동 적용.
- 비용: 발화 누락 사건이 실제 발생할 때까지 사용자 수동 운영 부담 지속.
- 보상: family 비용 감소 곡선 검증 (1→2 발행, 2→3 적용, 3→4 자동 인용 비용).

### (c) `decided-wontfix` — 발화 메커니즘 자체 폐기

- case #005 트리거 룰을 영구 수동으로 확정. 자동화 후보 제거.
- 비용: case #005가 정의한 자동화 지연 역설을 영구 수용. 본 레포 핵심 자동화 후보 1개 폐기.
- 가능성: 메모리 [멀티 에이전트 + 빠른 머지]는 자동화 선호 신호 — (c)와 반대 방향.

## 7. 트리거 임계점 (decided-observe 선택 시)

본 case는 *무엇을 더 봐야 하는지*를 명시한다 (case #005 §6 양식).

### 7.1 deadline-tracker 활성화 트리거

- **case 1건 이상이 deadline 경과했는데 회고 케이스가 미발행된 사건 1회 이상**
- case #010 deadline 2026-05-10이 첫 후보. 5/11에 회고 미발행 시 본 §7.1 임계점 충족
- case #005 §"마감일 도래" 룰의 *실측 위반* 자체가 활성화 입력

### 7.2 외부 사건 dispatch 활성화 트리거

- **광천샵에서 case #004/006/010-family 패턴이 발생했는데 본 레포에 케이스 미박제된 사건 2회 이상**
- 즉 *외부 사건 누락 비용*이 case 단위로 실측될 때
- case #002 후보 B(dispatch) + 후보 C(주기적 sweep) 동시 진입 후보

### 7.3 패턴 자동 감지 활성화 트리거

- §7.2가 먼저 충족된 후, dispatch만으로도 누락이 지속되어 *정적 감지*가 필요한 사건 1회 이상
- §7.3은 §7.2 의존 — 능동 감지는 수동 dispatch 후의 다음 단계

### 7.4 메타 룰

세 트리거 모두 **실제 발화 누락 사건**의 누적을 요구한다 — case #005 §6.1 "사실 박제 우선" 원칙. 본 case의 §7은 다음 결정 케이스 발행 시 직접 인용 가능한 형식으로 작성됨.
