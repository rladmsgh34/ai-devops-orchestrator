# Case 010 — 풀 루프 액터 파이프라인 (결정 케이스)

**TL;DR.** 사용자가 "Gemini가 이슈 작성 → Claude Code가 개발 → GitHub Actions가 CI/CD → Gemini가 최종 검토 → Claude Code가 배포" 풀 루프로 광천샵을 자율 개선하자고 요청. 이 요청을 곧장 7개 컴포넌트(기획·검증자 교체·배포 오케스트레이터·트리아지·컨텍스트 패커·승인 상태머신·롤백)로 코드화하면 case #007(외부 Unbacked Self-Claim) → case #009(내부, services/+docker-compose+INTEGRATIONS.md) family의 **세 번째 인스턴스**로 박제된다 (case #005 트리거 룰 재발동). 결정은 (b) `decided-observe` — 풀 루프 *설계*(액터 매핑·게이트 정의·트리거 임계점)는 본 케이스에 박제하되, **각 컴포넌트의 코드화는 그 컴포넌트가 요구되는 실제 사건이 누적된 후**에만 진입한다. 이는 case #009 결정(부분 실체화 + 격리)의 첫 자기 적용이다.

> case #009의 결정 룰("케이스 없이 컴포넌트를 박제하지 않는다, 첫 케이스가 요구할 때만 활성화")이 본 case의 사용자 요청에 대한 결정 룰을 이미 제공한다. 본 case는 그 룰의 **첫 자기 적용**이며, family 3차 인스턴스화를 *결정으로 회피*하는 경로를 박제한다.

## 1. 사용자 요청과 발생 맥락

2026-05-03 사용자가 ai-dev-loop-analyzer 정리 작업(PR #45) 직후 다음을 제기:

> "백그라운드에서 계속 작업이 가능할까? 이슈는 gemini cli가 작성하고 개발은 claude code가 하고 github로 ci cd 를 하고 최종 검토는 gemini cli가 하고 claude code 가 배포 하는 프로세스로 계속 광천샵을 개선할 수 있을까"

이어 "ai devops를 활용하면 될 것 같다", 결정 충돌 발견 후 "추천으로 진행해줘"로 발전 — 추천 전환은 *case #009 결정과의 정합성* 문제가 식별된 직후 이뤄짐.

발생 맥락의 본질:
- 사용자 요청은 **가설**(이런 루프가 가능할까) 단계이며 *실제 발생한 사건*에 의해 트리거된 것이 아님
- 풀 루프 7개 컴포넌트 중 **현재 시점에 어느 하나도 "케이스 없이는 만들지 않는다"는 case #009 결정을 우회할 근거가 없음**
- 단, 사용자 요청 자체는 **정직하게 박제할 가치가 있음** — 미래 실체화 진입의 설계 출발점이 됨

### 결론

본 case는 case #007/009 family의 **세 번째 인스턴스 후보**다. 차이점은 *발견 시점*에 있다 — case #007/009는 "이미 박제된 자산"을 사후에 발견했지만, 본 case는 **박제 행위 자체가 일어나기 전에 발견**되었다. 이 시점 차이가 결정의 비용을 결정적으로 낮춘다:

- 코드를 짠 뒤 회수(case #007의 archive, case #009의 profile 격리)하는 비용 < 코드를 짜기 전에 박제만 하고 보류하는 비용
- 따라서 본 case는 family 3차 인스턴스화를 *결정으로 회피*할 수 있는 첫 사례
- case #009의 결정 룰("첫 케이스가 요구할 때만 활성화") = case #005 트리거 룰의 입력. 본 case는 그 룰을 *실체화 진입 직전에* 적용

### 자기 검증의 메타 학습

case #006이 case #004에 대해, case #009가 case #007에 대해 자기 검증을 수행했듯, 본 case는 case #007/009 두 인스턴스에 대해 자기 검증을 수행한다. **family가 3차 인스턴스를 만나면, 결정 룰 자체가 직접 인용 가능한 검증자로 작동한다** — 즉 case #005 트리거 룰의 발동 비용이 family 깊이에 따라 *감소*한다 (1→2: 룰 발행 비용, 2→3: 룰 적용 비용, 3→4: 룰 자동 인용 후보).

## 6. 트리거 임계점 (decided-observe의 핵심)

본 case는 *무엇을 더 봐야 하는지*를 명시해야 한다 (case #005 §6 양식).

§2의 풀 루프 5단계 중 각 컴포넌트의 코드화 진입은 다음 임계점 충족 시 별도 결정 케이스로 발행한다.

### 6.1 단계 0 (기획용 gemini-cli) 활성화 트리거

- **사용자가 광천샵 작업 시작 시 "무엇을 할지" 단계에서 헛돈 사건이 누적 2회 이상**
- 또는 동일 회귀 도메인에 대한 fix PR이 발생한 직후 그 도메인의 다음 회귀 방지 이슈가 *생성되지 않은 채 같은 도메인 fix PR이 재발*한 사건 2회 이상
- 활성화 시 동반 갱신: README의 액터 다이어그램에 "단계 0 기획" 추가

### 6.2 검증자(gemini-cli, Antigravity 교체) 활성화 트리거

## 7. 마감일 도래 시

본 case는 §5에서 (b) `decided-observe`로 즉시 결정되었으므로 deadline은 **트리거 임계점의 첫 충족 시까지**가 실질적 의미를 가진다. 임계점 충족 없이 발행일로부터 2026-05-10까지 새 사건이 0건이면, 그 사실 자체가 사용자 요청이 *가설 단계*였음을 §1의 진단을 사후 검증한다 — 별도 회고 케이스 불필요 (정상 작동의 박제).

## 8. 학습 (ChromaDB 인덱싱 대상)

case #007/009 family는 **3차 인스턴스 후보를 *발생 직전에* 결정으로 회피한 첫 사례**를 본 case에서 얻는다. case #007이 외부 변형(이미 박제된 위성 repo의 archive), case #009가 내부 변형(이미 박제된 services/+docker-compose의 격리)이라면, 본 case는 **박제 행위 자체가 일어나기 전 변형** — 코드를 짜기 *전에* 박제만 하고 보류하는 변형. 이 시점 차이가 회수 비용을 0에 가깝게 만든다 (회수할 자산이 아직 없으므로).

family 깊이에 따라 결정 적용 비용은 감소한다:
- 1차 (#007): 룰 발행 비용 (case #005 §6 자체를 정의)
- 2차 (#009): 룰 적용 비용 (정의된 룰을 self-claim 자산에 처음 적용)
- 3차 (본 case): 룰 *자동 인용* — 결정 룰 자체가 검증자로 작동, 사용자 요청에 대한 결정 본문이 case #009 결정 룰의 직접 인용으로 도출됨
- 4차 이후 (가설): 룰이 코드화된 진입 게이트로 작동할 후보 — 본 case의 §6 트리거 임계점이 그 후보의 입력 형식

지휘자 모델의 **자기 적용 깊이**: case #006(case #004 자기 검증 1차) → case #009(case #007 자기 검증 2차) → 본 case(case #007/009 family 자기 적용 3차)로 누적된다. 깊이가 늘 때마다 *결정의 자동성*이 증가한다 — 본 case의 결정 §5 사유는 case #009 결정을 직접 인용함으로써 일부 자동 도출되었다. 다음 자기 적용(4차)은 *결정 본문이 룰 인용만으로 완전 도출*되는 단계가 될 후보 — 그 시점이 결정 케이스의 자동화 진입점.

### 본 case의 정직성 시그널

본 case는 *발생 직전*에 발견되어 회수할 자산이 0이라는 점에서 case #007(시그널 0)·case #009(시그널 부분)와 다르다. **시그널이 100%인 상태로 시작** — 사용자 요청·family 자기 검증·결정 사유가 본문 모두에 명시. 이 시점에서 family 자기 모순을 회피한 결정의 학습은, 미래 family 4차 후보가 발생할 때 *발생 직전 발견 → 결정으로 회피*가 표준 경로임을 박제한다.
