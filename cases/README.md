# Cases — 케이스 주도 개선 로그

이 폴더는 오케스트레이터의 **개선 근거 저장소**다.
실제 발생한 사건/요청만 기록하고, 그 기록이 코드 변경의 유일한 근거가 된다.

## 왜 이렇게 하는가

오케스트레이터를 처음부터 거대하게 설계하면:
- 쓸모없는 기능을 만들게 된다
- 실제 워크플로우와 어긋난다
- 유지보수 비용이 가치를 초과한다

대신 **실제 케이스가 발생할 때마다 최소한의 변경을 가하면**:
- 모든 코드 줄에 발생 근거가 있다
- 사용 안 되는 컴포넌트가 생기지 않는다
- 아키텍처가 현실에 맞춰 진화한다

## 무엇을 케이스로 기록하는가

| 기록함 ✅ | 기록 안 함 ❌ |
|---|---|
| 실제 발생한 운영 사고 | 가설적 시나리오 |
| Claude Code가 같은 실수 반복 | "이런 게 있으면 좋겠다" |
| Antigravity가 놓친 회귀 | 일반적 베스트 프랙티스 |
| main 직접 푸시 시도 | 외부 블로그에서 본 패턴 |
| 검증 누락 PR | 다른 회사 사례 |
| 사용자 승인 우회 시도 | 미래 트렌드 |
| 컨텍스트 부족으로 헛수고한 작업 | 일회성 단순 버그 |

## 케이스 작성 절차

1. `cases/TEMPLATE.md` 복사
2. 파일명: `NNN-<짧은-슬러그>.md` (예: `001-prisma-migration-rollback.md`)
3. 번호는 순차 증가, 최신 번호는 `git ls-files cases/ | sort | tail -1`로 확인
4. 즉시 커밋 (사실 기록은 빠를수록 정확함)
5. 케이스가 코드 변경을 요구하면 별도 PR로 진행하고, 그 PR이 이 케이스 파일을 참조

## 사건 발행 → 케이스 변환 절차 (case #002 후보 A)

외부 사건(gwangcheon-shop의 CI 실패, Antigravity가 놓친 검증, 사용자 피드백 등)은 다음 입구로 들어옵니다:

1. **issue 발행**: GitHub issue 템플릿 `Case Report (사건 발행)` 사용 (라벨 `case-candidate` 자동 부착)
2. **변환**: orchestrator 측에서 issue 내용을 위 "케이스 작성 절차"대로 `cases/NNN-*.md` 파일로 변환
3. **링크**: 변환 PR이 그 issue를 `Closes #<번호>`로 닫음
4. **인덱스**: 본 README 하단 인덱스에 행 추가

> 이 절차는 case #002 후보 A의 최소 구현입니다. 후보 B(자동 dispatch)·C(주기적 sweep)는 후속 케이스에서 추가됩니다.

## 케이스에서 코드로 가는 흐름

```
실제 사건 발생
  ↓
cases/NNN-*.md 작성 (사실만)
  ↓
"이 케이스가 반복될 가능성이 있는가?"
  ├─ No → 기록만 남기고 종료
  └─ Yes → "오케스트레이터가 어느 레이어에서 막을 수 있는가?"
            ↓
          최소 구현 PR (케이스 ID 참조)
            ↓
          유사 케이스 발생 시 같은 컴포넌트 확장
```

## 케이스 인덱스 유지

- 신규 케이스 작성 시 이 README 하단의 인덱스에 한 줄 추가
- 해결된 케이스는 인덱스에서 ✅ 표시
- 보류/관찰 중인 케이스는 ⏳
