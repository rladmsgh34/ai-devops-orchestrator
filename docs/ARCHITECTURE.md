# Architecture — 지휘자(Conductor) 모델

> 이 문서는 오케스트레이터의 **재정의된 포지션**을 기록한다.
> 기존 README의 "AI 6개 에이전트가 자동 수정" 모델은 폐기되었다.

## 1. 핵심 통찰

기존 모델은 오케스트레이터가 **4번째 개발자**처럼 동작했다. 그러나 실제 운영 환경의 액터는 이미 셋이 정해져 있다:

```
Claude Code  →  Antigravity  →  사용자
   (생성)        (검증)         (승인)
```

오케스트레이터가 또 다른 생성자/검증자로 끼면 **이중 작업**이 되거나 **검증을 우회**한다. 따라서 포지션을 **지휘자**로 재정의한다 — 코드를 만들지 않고, 셋 사이의 컨텍스트·상태·학습을 관리한다.

## 2. 지휘자가 하는 일 (5개 레이어)

```
┌─────────────────────────────────────────────────────────┐
│                    사용자 (승인자)                       │
└──────────────────────────▲──────────────────────────────┘
                           │ (5) 승인 게이트
┌──────────────────────────┴──────────────────────────────┐
│                  Antigravity (검증자)                    │
└──────────────────────────▲──────────────────────────────┘
                           │ (3) 컨텍스트 패킹 / (4) 검증 게이트
┌──────────────────────────┴──────────────────────────────┐
│                 Claude Code (생성자)                     │
└──────────────────────────▲──────────────────────────────┘
                           │ (1) 컨텍스트 주입 / (2) 트리아지
┌──────────────────────────┴──────────────────────────────┐
│              Runtime (스테이징 / 운영)                   │
└─────────────────────────────────────────────────────────┘
                           ▲
                           │
              ┌────────────┴────────────┐
              │  ChromaDB (학습 저장소)  │
              └─────────────────────────┘
```

### Layer 1 — 컨텍스트 주입 (Pre-Generation)
Claude Code가 작업을 시작할 때, 해당 파일/도메인의 **과거 사고 사례**를 ChromaDB에서 조회해 프롬프트에 자동 첨부.
- 입력: 작업 대상 파일 경로, 이슈 텍스트
- 출력: 관련 케이스 N건 요약 + 권장 가드레일

### Layer 2 — 트리아지 라우팅 (Inbound Routing)
알림(에러 로그, 알람, 사용자 피드백)이 들어오면 분류해 올바른 액터에게 보냄:
- "Claude Code 재작업감" — 코드 변경 필요
- "Antigravity 재검증감" — 기존 코드의 검증 누락/회귀
- "운영/인프라 이슈" — 코드와 무관, 오케스트레이터가 직접 처리
- "사용자 결정 필요" — 정책/우선순위 판단

### Layer 3 — 컨텍스트 패킹 (Generation → Verification)
Claude Code가 PR을 만들면, Antigravity 검증 요청 시 자동 번들 생성:
- 변경된 파일 + diff
- 영향받는 테스트 목록
- 관련 도메인의 과거 회귀 사례
- 의심 영역(예: "이 파일은 작년에 race condition이 있었음")

### Layer 4 — 검증 게이트 (Verification → Approval)
Antigravity 검증이 끝났는지 확인. 검증 누락된 PR은 사용자 승인 단계로 못 넘어감.
- main 직접 푸시 차단 (CLAUDE.md 룰의 코드화)
- 검증 결과 부재 / 실패 시 자동 차단

### Layer 5 — 승인 게이트 + 배포 (Approval → Deployment)
사용자 승인 후에만 배포 트리거. 카나리 → 헬스체크 → 자동 롤백.
- 운영 레이어라 액터 충돌 없음, 기존 구조 유지
- 운영 에러 발생 시 → ChromaDB에 기록 → Layer 1으로 환류

## 3. 컴포넌트 매트릭스

| 컴포넌트 | 기존 모델 | 새 모델 | 비고 |
|---|---|---|---|
| Error Pattern Analyzer | AI가 분석·수정 PR | **유지(분석만)** | 수정 PR 생성 제거, 분석 결과는 Claude Code 컨텍스트로 |
| Security Vulnerability Scanner | 자체 스캔 | **제거** | Antigravity로 위임 |
| Code Quality Enforcer | 자체 룰 엔진 | **제거** | Antigravity로 위임 |
| Performance Regression Detector | 자체 감지 | **유지** | 운영 메트릭 기반, 액터 충돌 없음 |
| Infrastructure Monitor | 24/7 모니터링 | **유지** | 운영 레이어 |
| Deployment Orchestrator | 배포 자동화 | **유지 + 강화** | 승인 게이트 추가 |
| **컨텍스트 패커** | 없음 | **신규** | Layer 1, 3 |
| **승인 상태머신** | 없음 | **신규** | Layer 4, 5 |
| **트리아지 라우터** | 없음 | **신규** | Layer 2 |
| **케이스 로그** | 없음 | **신규** | 개선의 근거 |
| ChromaDB | 학습 저장소 | **유지 + 확장** | 환류 루프 명확화 |
| n8n 워크플로우 엔진 | 자동화 허브 | **유지** | 신규 레이어들이 위에서 동작 |

## 4. gwangcheon-shop 적용 시나리오

### 시나리오 A — 새 기능 개발
1. 사용자: "결제 페이지에 쿠폰 적용 기능 추가해줘"
2. **Layer 1**: 오케스트레이터가 `결제`, `쿠폰`, `Prisma 스키마 변경` 관련 과거 케이스를 ChromaDB에서 조회 → Claude Code 프롬프트에 첨부
3. Claude Code: 작업 → PR 생성
4. **Layer 3**: 오케스트레이터가 변경 파일·테스트·과거 회귀 사례 번들 → Antigravity에 전달
5. Antigravity: 검증 결과 게시
6. **Layer 4**: 검증 결과 확인 → 통과 시 사용자 승인 단계로
7. 사용자: 승인
8. **Layer 5**: 스테이징 배포 → 헬스체크 → 운영 배포
9. 운영 에러 발생 시 → ChromaDB 기록 → 다음 작업 시 Layer 1에서 자동 첨부

### 시나리오 B — 운영 에러
1. 운영에서 에러 발생 (예: 결제 실패율 급증)
2. **Layer 2**: 오케스트레이터가 분류
   - 코드 회귀 의심 → "Claude Code 재작업감" + 컨텍스트 번들
   - 인프라 문제 → "운영 이슈" 직접 처리 + 사용자 알림
   - 외부 의존성 → "사용자 결정 필요" 알림
3. 분류된 액터로 라우팅 → 시나리오 A 흐름 재진입

## 5. 케이스 주도 개선 원칙

이 아키텍처는 **완성형이 아니다**. `cases/`에 실제 사례가 쌓일수록 각 레이어의 구현이 구체화된다.

- 새 컴포넌트는 케이스 ≥1건 후에만 코드화
- 기존 컴포넌트는 케이스가 한계를 드러낼 때만 확장
- 가설 기반 기능 추가 금지 — `cases/`에 근거 없으면 만들지 않음

자세한 절차는 [`cases/README.md`](../cases/README.md) 참조.
